# ywtechit-stock-bot

> 💡 **"디테일한 전략을 주면, LLM이 사람보다 매매를 잘할까?"**
>
> 이 질문을 직접 검증하기 위해 개발한 **한국투자증권(KIS) Open API 기반 자동매매 봇**입니다. LLM에게 시세·잔고 스냅샷과 전략 명세를 주고 판단을 내리게 하되, 손절·한도·거래 횟수 같은 **위험 통제(Risk Control)는 코드가 직접 강제**합니다. 모든 판단과 주문은 SQLite에 기록되어 추후 "LLM이 단순 Buy & Hold를 이겼는지" 분석할 수 있습니다.

## 🎯 핵심 컨셉: 운전자와 브레이크의 분리

LLM은 시장 맥락을 읽고 유연하게 판단하는 데 강하지만, 이를 검증 없이 주문으로 연결하는 것은 위험합니다(환각, 수량 오류, API 오류 등). 따라서 본 프로젝트는 역할을 명확히 이원화합니다.

* **LLM (운전자)**: 무엇을 살지/팔지 **제안**합니다. (실행 권한 없음)
* **가드레일 (브레이크)**: 모든 제안을 주문 직전 **검문**하여 위반 시 차단하거나 수량을 깎습니다. **손절은 LLM을 거치지 않고 코드가 즉시 실행**합니다.

## ⚙️ 2단 의사결정 구조 (Architecture)

봇은 LLM의 판단을 그대로 신뢰하지 않습니다. 코드 가드레일이라는 두 번째 관문을 통과해야만 실제 주문이 거래소로 나갑니다.

### 왜 코드가 일부 강제성을 지닐까?

LLM은 시장 맥락을 읽고 유연하게 판단하는 데는 강하지만, 그 판단을 **그대로 주문으로 연결하는 것은 위험**합니다. 환각으로 엉뚱한 수량을 부르거나, API 오류를 빈 응답으로 흘리거나, 같은 주문을 반복할 수 있기 때문입니다. 그래서 이 봇은 역할을 명확히 둘로 나눕니다.

- **LLM (운전자)** — 무엇을 살지/팔지 *제안*한다. 단, 제안일 뿐 실행 권한은 없다.
- **가드레일 (브레이크)** — 모든 제안을 주문 직전에 *검문*하고, 위반하면 차단하거나 한도 내로 깎는다. 손절은 아예 LLM에게 묻지도 않고 코드가 즉시 실행한다.

자율주행에 비유하면 LLM이 핸들을 잡되, 차선이탈·급발진을 막는 브레이크는 코드가 따로 쥐고 있는 구조입니다.

## ⚙️ 2단 의사결정 구조 (Architecture)

봇은 LLM의 판단을 **그대로 신뢰하지 않습니다.** LLM이 무엇을 고르든 그것은 *후보*일 뿐, 코드 가드레일이라는 두 번째 관문을 통과해야만 실제 주문이 거래소로 나갑니다. 아래 다이어그램이 한 종목이 결정에서 주문까지 거치는 전체 경로입니다.

```
                  ┌─────────────────────────────────────────────┐
                  │                  메인 루프                   │
                  │              (main.py / 장중 반복)            │
                  └─────────────────────────────────────────────┘
                                       │
            ┌──────────────────────────┼──────────────────────────┐
            ▼                          ▼                          ▼
   ┌─────────────────┐       ┌──────────────────┐       ┌──────────────────┐
   │  ① 하드 손절    │       │  ② LLM 판단      │       │  ③ 가드레일 검문 │
   │  (LLM 무시)     │       │  BUY/SELL/HOLD   │  ───▶  │  (코드 강제)     │
   │  즉시 청산      │       │  + 수량 + 근거   │       │                  │
   └────────┬────────┘       └──────────────────┘       └────────┬─────────┘
            │                                                     │
            │                      ┌──────────────────────────────┤
            │                      ▼ 통과                  위반 ▼  │
            │           ┌──────────────────┐       ┌──────────────────────┐
            └──────────▶│  KIS 주문 전송   │       │  차단 / 한도 내 축소  │
                        │  (adapter)       │       │  / HOLD              │
                        └────────┬─────────┘       └──────────┬───────────┘
                                 │                            │
                                 ▼                            ▼
                        ┌─────────────────────────────────────────────┐
                        │   기록(SQLite Journal) + 알림(Discord)       │
                        └─────────────────────────────────────────────┘
```

다이어그램에서 가장 중요한 두 갈래는 이것입니다.

- **① 하드 손절은 LLM을 거치지 않습니다.** 보유 종목이 손절선에 닿으면 곧바로 청산 주문으로 직행합니다. "팔까요?"라고 물을 여유가 없는, 가장 우선순위 높은 안전장치이기 때문입니다.
- **③ 가드레일을 통과한 주문만** KIS로 나갑니다. 위반하면 차단(HOLD)되거나 한도 내 수량으로 축소되며, 그 사실까지 모두 기록·알림됩니다.

### 가드레일이 강제하는 규칙

LLM이 무엇을 고르든, 주문 직전에 [risk/guardrails.py](risk/guardrails.py)가 아래를 무조건 적용합니다. 외부 의존성이 없는 순수 함수라 크리덴셜 없이도 단위테스트로 검증할 수 있습니다([tests/test_guardrails.py](tests/test_guardrails.py)).

| 규칙 | 동작 | 관련 설정 |
|------|------|-----------|
| **하드 손절** | 보유 종목 손익률이 한도 이하면 LLM에 묻지 않고 즉시 청산 | `HARD_STOP_LOSS_PCT` |
| **Kill switch** | 당일 실현손실이 한도 도달 시 당일 신규 매매 전면 중단 | `DAILY_MAX_LOSS_KRW` |
| **1회 매수금액 한도** | 초과분만큼 수량 자동 축소 (0주가 되면 차단) | `MAX_ORDER_KRW` |
| **종목당 최대 비중** | 총평가금액 대비 비중 초과분 축소 | `MAX_POSITION_PCT` |
| **가용현금 한도** | 주문가능 현금으로 살 수 있는 만큼만 축소 | — |
| **하루 최대 거래 횟수** | 당일 누적 주문이 한도 도달 시 차단 | `MAX_TRADES_PER_DAY` |
| **멱등성** | 같은 `(종목, 방향)` 주문이 직전 사이클에 이미 나갔으면 중복 차단 | — |
| **장 운영시간** | KRX 정규장(평일 09:00–15:30) 외에는 차단 | — |

## 아키텍처

### 계층 구조

코드는 책임에 따라 4개 계층으로 나뉘고, 의존 방향은 **위(오케스트레이션)에서 아래(외부 세계)로만** 흐릅니다. 덕분에 각 계층을 독립적으로 이해하고 테스트할 수 있습니다.

```
┌──────────────────────────────────────────────────────────────────────┐
│  오케스트레이션          main.py                                       │
│  장중 루프를 돌리며 아래 계층을 조율. "언제 무엇을 호출할지"만 안다.    │
└───────────────┬──────────────────────────────────────────────────────┘
                │
┌───────────────┼──────────────────────────────────────────────────────┐
│  의사결정      ▼                          판단                  검문     │
│        strategy/context.py        strategy/llm_decider.py   risk/guardrails.py
│        스냅샷·전략 명세 구성  ──▶  LLM이 BUY/SELL/HOLD 제안 ──▶ 코드가 최종 검문 │
│                                                            (순수 로직)  │
└───────────────┬──────────────────────────────────────────────────────┘
                │  정규화된 dict/list만 주고받음 (KIS 응답 포맷을 위로 노출하지 않음)
┌───────────────┼──────────────────────────────────────────────────────┐
│  어댑터        ▼                                                       │
│        adapter/broker.py              adapter/market_stream.py         │
│        REST: 시세·잔고·주문·체결       WebSocket: 실시간 체결가         │
│        (try/catch·재시도·throttle)    (별도 스레드·자동 재연결)        │
└───────────────┬──────────────────────────────────────────────────────┘
                │
┌───────────────┼──────────────────────────────────────────────────────┐
│  부가          ▼                                                       │
│        store/journal.py (SQLite 기록)   notify/discord.py (알림)       │
│        config.py + kis_bootstrap.py (설정·인증 부트스트랩)             │
│        vendor/kis/ (공식 KIS 샘플 — 수정 금지)                         │
└──────────────────────────────────────────────────────────────────────┘
```

핵심 설계 원칙은 **"KIS API의 지저분함을 어댑터 계층에 가둔다"** 입니다. 모든 재시도·throttle·재인증·응답 파싱은 [adapter/broker.py](adapter/broker.py) 안에서만 일어나고, 위 계층은 `{code, price, ...}` 같은 깔끔한 dict만 받습니다. 그래서 가드레일과 전략 로직은 KIS 응답 포맷을 전혀 몰라도 됩니다.

### 한 사이클의 흐름

`main.py`의 메인 루프가 장중 반복하는 한 사이클은 다음 6단계입니다.

```
 1. 스냅샷 구성      잔고·포지션·현재가 조회         (strategy.context.build_snapshot)
 2. 하드 손절 점검   손절선 도달 종목 즉시 청산       (risk.guardrails.force_stop_loss)  ← LLM 무시
 3. LLM 판단         종목별 BUY/SELL/HOLD 결정       (strategy.llm_decider.decide)  ← N분 간격으로만
 4. 가드레일 검문    한도·횟수·중복 검증/축소         (risk.guardrails.check)
 5. 주문 전송        통과 건만 시장가 주문           (adapter.broker.place_order)
 6. 기록 + 알림      결정·주문 저장 / 디스코드 통보   (store.journal / notify.discord)
```

타이밍과 안전성에 대한 두 가지 설계 결정:

- **호출 주기를 분리합니다.** 하드 손절(2)은 매 사이클(약 10초)마다 점검해 빠르게 반응하지만, 비용이 드는 LLM 판단(3)은 `DECISION_INTERVAL_MIN`(기본 15분) 간격으로만 호출합니다. 매 틱마다 LLM을 부르지 않아 API 비용과 노이즈를 줄입니다.
- **무인 실행을 전제로 실패를 흡수합니다.** 모든 외부 호출(KIS·LLM·Discord)은 `try/catch`로 감쌉니다. LLM 응답이 스키마를 벗어나거나 API가 죽으면 **안전하게 HOLD로 폴백**하고, 스냅샷 조회가 실패하면 잠시 쉬었다 재시도하며, 메인 루프에서 미처리 예외가 나면 디스코드로 경고한 뒤 포지션을 건드리지 않고 안전 종료합니다. "조용히 멈춰서 망가지는" 상황을 막는 것이 목표입니다.

### LLM 출력을 강제하는 방법

LLM이 자유 텍스트로 "삼성전자가 좋아 보이니 사세요" 같은 답을 내면 파싱도 어렵고 위험합니다. 그래서 [strategy/llm_decider.py](strategy/llm_decider.py)는 Anthropic **tool use(function calling)** 로 출력 형식을 강제합니다.

- LLM은 반드시 `submit_decision` 도구를 호출해야 하며, 그 입력 스키마가 `{action, quantity, signal, rule, summary}`로 고정되어 있습니다. 자유 텍스트 매매는 원천 차단됩니다.
- `signal`(결정의 직접 트리거가 된 실제 수치)과 `rule`(근거한 전략 조항)을 함께 받아, 알림과 저널에 "왜 이 결정을 했는지"가 추적됩니다.
- 스키마를 벗어난 응답, 파싱 실패, API 오류는 전부 **HOLD로 폴백**합니다 — 의심스러우면 거래하지 않는 쪽이 안전하니까요.
- 시스템 프롬프트와 전략 명세는 매번 동일하므로 **prompt caching**을 적용해 반복 호출 비용을 줄입니다.

## 모듈 구성

```
kis-llm-trader/
├── main.py                  ← 장중 메인 루프 (스냅샷→판단→검문→주문→기록→알림)
├── config.py                ← .env 검증 + kis_devlp.yaml 런타임 생성
├── kis_bootstrap.py         ← config→vendor/kis 로드 순서를 강제하는 단일 진입점
│
├── strategy/
│   ├── context.py           ← 전략 명세(STRATEGY_SPEC) + 시세/포지션/현금 스냅샷
│   └── llm_decider.py       ← Anthropic tool use로 {action, quantity, 근거} 강제
│
├── risk/
│   └── guardrails.py        ← 코드 강제 안전장치 (외부 의존성 없는 순수 로직)
│
├── adapter/
│   ├── broker.py            ← KIS REST 래퍼 (시세/잔고/주문/체결)
│   └── market_stream.py     ← 실시간 체결가 웹소켓 + 자동 재연결
│
├── store/
│   └── journal.py           ← SQLite 기록 (decisions/orders/fills/daily_pnl)
│
├── notify/
│   └── discord.py           ← 디스코드 Webhook embed 알림
│
├── vendor/kis/              ← 공식 KIS 샘플 사본 (수정 금지)
└── reference/               ← 공식 레포 submodule (vendor 복사 원본)
```

| 모듈 | 역할 | 설계 포인트 |
|------|------|-------------|
| `config.py` | `.env` 검증 + 공식 kis_auth용 `~/KIS/config/kis_devlp.yaml` 생성 | 크리덴셜은 오직 `.env`에만. 필수 키 누락 시 기동 거부 |
| `kis_bootstrap.py` | config → vendor/kis 순서 강제 | kis_auth는 import 시점에 yaml을 읽으므로 순서가 중요 |
| `strategy/context.py` | 전략 명세 + 스냅샷 구성 | 사람이 직접 다듬는 전략 룰의 '의도'만 기술 |
| `strategy/llm_decider.py` | LLM 결정을 tool use로 강제 | 자유 텍스트 매매 금지. 파싱 실패·API 오류 → HOLD 폴백. prompt caching 적용 |
| `risk/guardrails.py` | 코드 강제 안전장치 | 외부 의존성 없는 순수 로직 → 크리덴셜 없이 단위테스트 가능 |
| `adapter/broker.py` | KIS REST 래퍼 | **모든 API try/catch·재시도·throttle를 여기서만**. 상위는 정규화된 dict만 받음 |
| `adapter/market_stream.py` | 실시간 체결가 웹소켓 | 별도 스레드 + supervisor로 끊기면 backoff 재연결 + 알림 |
| `store/journal.py` | SQLite 기록 | "AI가 buy&hold를 이겼나" 오프라인 분석용. 기록 실패가 매매를 막지 않음 |
| `notify/discord.py` | 디스코드 알림 | 일반 채널(중요 이벤트) + trade-log 채널(매 사이클 결정 전부) 분리 |

## 시작하기

### 1. 사전 준비 (코드 외 — 직접 발급)

1. **[KIS Developers](https://apiportal.koreainvestment.com/)** 에서 Open API 신청
   → **모의/실전 각각** 앱키·앱시크릿, HTS ID, 계좌번호
2. **디스코드** 채널 Webhook URL (가급적 일반/trade-log 2개 채널)
3. **Anthropic** API Key

### 2. 셋업

```bash
# 공식 KIS 레포 submodule 초기화 (clone 직후)
git submodule update --init --recursive

# 의존성 설치
uv sync

# 환경변수 설정
cp .env.example .env   # 편집해서 실값 입력 (.env 는 절대 커밋 금지)
```

주요 환경변수 (전체는 [.env.example](.env.example) 참고):

| 변수 | 설명 | 기본값 |
|------|------|--------|
| `KIS_ENV` | 실행 환경: `vps`(모의) / `prod`(실전) | `vps` |
| `WATCH_CODES` | 관심종목 코드(콤마 구분) | `005930` |
| `DECISION_INTERVAL_MIN` | LLM 의사결정 호출 간격(분) | `15` |
| `MAX_ORDER_KRW` | 1회 매수 최대 금액(원) | `100000` |
| `MAX_POSITION_PCT` | 종목당 최대 비중(총평가금액 대비) | `0.4` |
| `DAILY_MAX_LOSS_KRW` | 일일 최대 손실 → kill switch(원) | `30000` |
| `MAX_TRADES_PER_DAY` | 하루 최대 거래 횟수 | `5` |
| `HARD_STOP_LOSS_PCT` | 하드 손절선(%, 음수) | `-10.0` |

### 3. 실행
